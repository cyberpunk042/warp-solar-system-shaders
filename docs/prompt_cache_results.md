# Shared-prefix prompt cache — reproducible run record (the LLM experiment that matters most)

Generated by `python -m warp_compress.prompt_cache` (token store: Warp GPU; prefill: real gpt2). Two things
measured and kept **honestly separate**: (1) ChromoFold's contribution — the compact token store + GPU span
recovery; (2) the prefill/TTFT win, which is *prefix-KV sharing* (a standard technique this batch shape
enables), not ChromoFold.

## Part 1 — the token store (ChromoFold's actual contribution)

```
prompt-cache token store: K=256 requests, prefix=800, suffix=24 tokens each
[correct] GPU span recovery == prefix+suffix ✓
[store]  raw-duplicated 421.9 KB   ChromoFold shared-prefix 13.9 KB  => 30.4× smaller
[recover] 262,144 random span tokens in 0.92 ms  (3.5 ns/token, O(1) on the GPU)
```

The shared prefix is stored **once**; each request adds only its short suffix. So a batch of shared-prefix
requests is ~30× smaller than a duplicated prompt cache, which is ~30× more requests per VRAM budget — and any
request's token span reconstructs on the GPU in O(1) (3.5 ns/token).

## Part 2 — prefill / TTFT on real gpt2 (prefix-KV sharing)

```
prefill / TTFT on gpt2 (prefix=341 tok, 24 requests, suffix≈9 tok)
[prefill] naive (full prompt each) 10346.2 ms   prefix-shared (KV once + suffixes) 1760.3 ms  => 5.9× faster
[TTFT]    naive ≈ 431.1 ms/request   prefix-shared ≈ 57.1 ms/request (suffix-only forward on the shared KV)
```

Computing the shared-prefix KV once and reusing it across the 24 requests cuts prefill 5.9× and TTFT 7.5×.
**This win is prefix-KV caching, not ChromoFold** — but ChromoFold's compact token store + O(1) span recovery
is what makes holding and routing such a batch cheap. Together: more shared-prefix requests resident, each
prefilled by reusing one shared prefix. No claim to replace the KV cache or compress dense weights.

# The validating capability table

Assembled from the measured runs (`bench_gpu_results.md`, `bench_frontier_results.md`, this file). "code
(gpt2)" corpus, V=50257, one RTX 2080 Ti, PCIe 3.0 ×16. Latency figures are for *serving a q-token random
slice* at scale; ✓/✗ are capabilities, not speed.

| Representation      | Resident B/tok | Random 1-tok | 1K-tok slice | Search | Whole decode | CPU round-trip |
|---------------------|----------------|--------------|--------------|--------|--------------|----------------|
| raw uint16          | 2.00           | GPU gather µs | GPU gather µs | ✗      | free (is raw)| none (resident) |
| raw uint32          | 4.00           | GPU gather µs | GPU gather µs | ✗      | free (is raw)| none (resident) |
| gzip                | 0.70           | ✗ (decompress-all ≈348 ms) | ✗ (same) | ✗ | 348 ms | yes (CPU codec) |
| zstd-19             | 0.63           | ✗ (decompress-all ≈6 ms) | ✗ (same) | ✗ | 6 ms | yes (CPU codec) |
| packed wavelet      | 2.25           | ~kernel µs   | ~kernel µs   | rank   | GPU          | no (GPU-resident) |
| RRR wavelet (FM)    | 1.14           | ~kernel µs   | ~kernel µs   | rank + count/locate/predict | GPU | no |
| full ChromoFold¹    | 1.14 + SA      | ~kernel µs   | ~kernel µs   | count/locate/predict | GPU | no |
| shared-prefix store² | prefix once + suffixes | 3.5 ns/tok | 3.5 ns/tok | — | GPU O(1) | no |

¹ RRR self-index + C array + a sampled suffix array for `locate` (`sa_sample` tunable — memory⇄latency).
² Not a general codec — the prompt-cache structure (one shared prefix + K suffixes); 30× vs duplicated here.

**Reading it:** for the *whole* stream, zstd is both smaller and fast to decode — use it for archival. For
**sparse random reads + search on a large, GPU-resident store**, only the ChromoFold rows serve a slice without
a full decode and without a CPU round-trip, at ~µs kernel latency scaling with q (68–111× vs zstd-decompress-
all for q ≤ 16K on a 4M-token stream; see `bench_frontier_results.md`). That intersection — sparse + searchable
+ GPU-resident + at scale — is the niche, and the shared-prefix prompt cache sits squarely in it.
