# ChromoFold compression frontier — reproducible run record (agenda E + F)

Generated by `python -m warp_compress.bench_frontier` (gpt2 tokenizer, V=50257; zstd level 19). A snapshot for
interpretability; re-run on your data. **This is the decisive positioning result, and it is honest: ChromoFold
is not a ratio play against LZ — its value is random access + search at scale.**

raw = uint16 fixed width · packed = wavelet-of-sequence (access+rank) · RRR-FM = RRR wavelet-of-BWT
(access + rank + **search**, entropy-sized) · gzip/zstd = whole-stream, **no random access**.

## Random-access latency at scale (4M-token stream, uint16 = 8 MB)

gzip/zstd have no random access — serving **any** slice requires decompressing the whole 8 MB
(gzip ≈ 348 ms, zstd ≈ 6.1 ms, constant). The ChromoFold access kernel scales with the query size:

```
          q  ChromoFold kernel  vs zstd-all
          1            55.3 µs        111×
          8            71.6 µs         85×
         32            69.6 µs         88×
        128            75.8 µs         81×
       1024            77.9 µs         79×
      16384            89.8 µs         68×
     262144           507.6 µs         12×
    4000000          7136.6 µs          1×  (all)
```

## The honest three-part conclusion

1. **Ratio: LZ wins.** gzip/zstd beat the FM-index self-index on every corpus — dramatically on redundant ones
   (sys-prompt: zstd 0.02 vs RRR-FM 0.69), because LZ copies long exact repeats the BWT self-index does not.
   ChromoFold's RRR-FM beats raw/packed (1.1–1.7 vs 2.0–2.25 B/tok) and on uniform noise loses even to raw.
   **ChromoFold is not a compression-ratio product.**
2. **Random access at scale: ChromoFold wins, ~1–2 orders of magnitude.** On the 4M stream a sparse read (q ≤
   16K) is 68–111× faster than a zstd whole-stream decompress, and the cost scales with q while decompress-all
   is fixed. (The ~55 µs floor at q=1 is fixed kernel-launch/event overhead, amortized by batched serving.)
3. **Search: only ChromoFold has it.** `count`/`locate`/`predict_next` run in the compressed domain; zstd
   can't answer them at all.

**Niche (honest):** large, GPU-resident token stores with *sparse random reads + search* — KV/context memory,
RAG retrieval, prompt-cache span recovery — not archival ratio. For pure archival, use zstd. The next
experiment (shared-prefix prompt cache) is chosen precisely because it exercises sparse span-recovery at
scale, where this niche pays.
