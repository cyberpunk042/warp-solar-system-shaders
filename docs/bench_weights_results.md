# Entropy-coding quantized weights — reproducible run record (the "combine with quantization" thesis)

Generated by `python -m warp_compress.bench_weights` (real gpt2, RRR wavelet over quantized weights). Validates
that quantized model weights — peaky histograms — compress further, **losslessly**, on top of quantization,
with GPU random access. Quantization is the lossy lever; this entropy layer recovers the exact quantized
values, so a forward pass is byte-identical to the plain-quantized model.

## Reading it (honest)

- **Real win on int4.** Entropy-coding quantized weights beats fixed int4 by **1.78× overall** (up to 2.99× on
  the MLP proj), because quantized weights cluster near zero. Composed with int4's 4× vs fp32, that's ~7× vs
  fp32 — and every weight stays **randomly addressable on the GPU** (fetch the routed MoE expert / dequant on
  the fly).
- **Lossless on top of quantization.** `max|Δlogits| = 0.00` between the RRR-stored and plain-quantized model:
  the entropy layer recovers the *exact* quantized values. (The 18.19 quant-vs-fp32 gap is int4 quantization's
  own cost — a quantization concern, not ChromoFold's; use a gentler quantizer if that matters.)
- **int8's margin is thin (1.18×)** — with 8 bitplanes the RRR **class-stream floor** (~0.27 b/bit × planes)
  eats most of the entropy savings. Compressing the class stream is the identified next lever; it would lift
  int8 and push int4 toward its H0 (≈1.2 b/w on the peakiest tensors).

## The weights stratum of "fit more on one GPU"

This is the concrete, measured version of docs/chromofold.md §4/§5: quantization × a lossless entropy layer,
per-weight GPU-addressable. Store many layers / MoE experts entropy-coded in VRAM, unfold the routed one on
demand — the honest, buildable path, not a claim to compress dense fp32 weights directly.
