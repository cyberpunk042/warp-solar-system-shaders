## The honest conclusions

1. **Ratio: xz wins on both payloads.** weights 1.14 < gz 1.22 < cfold 1.60; tokens 4.96 < 5.82 < cfold 9.25.
   **ChromoFold is not a ratio play against xz** вЂ” its edge is O(1)/O(log) **random access + search**, which
   gz/xz cannot do at all (any single value costs a whole-stream decompress).
2. **Stacking is not free, and not zero-gain** (correcting the earlier claim that cfold's payload is "already
   entropy"): `cfoldв†’xz` shrinks the blob a lot (weights 1.60в†’1.18, tokens 9.25в†’7.35) because the container's
   **superblocks** (monotone cumulative sums), Huffman/scale tables, and JSON header are very compressible. But
   it costs real time **and destroys random access** вЂ” so it's only for **cold archive**, and there xz-direct is
   both smaller (1.14 vs 1.18) and simpler. Never wrap a live cfold index.
3. **Order (the hypothesis, tested): cfold is terminal.** A stream compressor *before* it has nothing to consume
   (cfold ingests raw values/tokens, not bytes); *after* it, it recovers a little ratio but defeats the purpose.
   So the "stack" is a **choice, not a chain**: **cfold when you need GPU random access / search, xz when you
   need max cold-archive ratio** вЂ” not both.
4. **Optimization lead surfaced by this comparison вЂ” now implemented.** The ~0.4 b/val that xz recovered from
   the cfold blob is compressible index metadata (monotone superblocks / word bases). The container now
   delta+zlib-compresses those sections on serialization (they still decode to the same int32 arrays in VRAM,
   so **random access is untouched**): the gpt2 int4 weight blob dropped **1.601 в†’ 1.295 b/val** вЂ” now within
   ~13% of xz (1.14) and below gz (1.22), *while keeping O(1) GPU random access*. ChromoFold isn't chasing that
   last 13%; it's buying the random access and search xz can't offer. See `docs/chromofold_positioning.md`.
